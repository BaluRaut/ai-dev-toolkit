⚡ Get Started

Install & Configure

Clone, configure your API keys, and run your first workflow in under 2 minutes.

# 1. Clone the repository
git clone https://github.com/BaluRaut/ai-dev-toolkit.git
cd ai-dev-toolkit/claude-agent

# 2. Create virtual environment & install deps
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 3. Copy env template
cp .env.example .env
# Required
ANTHROPIC_API_KEY=sk-ant-...

# Optional integrations
JIRA_BASE_URL=https://yourorg.atlassian.net
JIRA_EMAIL=you@company.com
JIRA_API_TOKEN=<atlassian-token>

FIGMA_ACCESS_TOKEN=figd_...

# Analytics (optional)
NEXT_PUBLIC_AMPLITUDE_API_KEY=your_amplitude_key
# Interactive menu (recommended for first-time users)
python main.py

# Full workflow: Jira + Figma + Swagger to Code
python main.py --jira PROJ-123 --figma https://figma.com/file/abc --swagger ./api.json

# Review a file
python main.py --review src/components/UserCard.tsx

# Show all session costs
python main.py --costs

Full Workflow — Jira + Figma + API → Code
Generate a complete feature from all three sources in one shot
▾
python main.py --jira PROJ-123 --figma https://figma.com/file/abc123 --swagger ./api/swagger.json
⚙️ Under the Hood
STEP 01
Fetch Jira ticket via REST API
Pulls title, description, acceptance criteria, labels, story points from Jira Cloud v3.
STEP 02
Fetch Figma design + render screenshot
Downloads component tree JSON + renders a PNG via Figma API for Claude Vision. Base64-encoded and attached.
STEP 03
Parse OpenAPI / Swagger spec
Extracts endpoints, request/response schemas, auth headers. Optionally filters by --api-filter path prefix.
STEP 04
Compose mega-prompt with all context
Merges all three sources into a structured prompt with the Figma screenshot attached (Vision mode).
STEP 05
Claude generates multi-file output
Returns: component code, API service layer, Jest + RTL unit tests, Playwright E2E, TypeScript types, barrel exports.
STEP 06
Preview → confirm → write to disk
Shows a rich diff preview and prompts before writing. Token costs tracked automatically via TokenTracker.
📦 Outputs
Component.tsx Component.test.tsx e2e.spec.ts api.service.ts types.ts
💡 Use Cases
Sprint kickoff — dev picks up ticket, runs one command, gets full boilerplate Design handoff — PM shares Figma link, agent generates pixel-accurate React component New microservice — swagger spec → full service layer + integration tests automatically

Self-Healing Test Loop
Run tests → AI fixes failures → repeat until all pass
▾
python main.py --heal ./my-react-app --heal-framework jest --heal-retries 5
⚙️ Under the Hood
STEP 01
Run the full test suite via subprocess
Executes npx jest / vitest run / playwright test / pytest. Captures stdout, stderr, and exit code.
STEP 02
Parse failing test names and error output
Extracts structured failure objects: test name, assertion that failed, and stack trace line number.
STEP 03
Send failures + source files to Claude
Context includes: failing test, source file being tested, and all previous fix attempts to prevent infinite loops.
STEP 04
Apply Claude's fix to disk
Writes patched files. Records attempt history with timestamps and diff summaries.
STEP 05
Re-run suite → loop until green or max retries hit
Terminates on all-pass or max-retries exceeded. Prints a per-attempt healing report with pass/fail counts.
💡 Use Cases
After a major refactor — fix all broken tests in one command Dependency upgrade — new library version breaks 30 tests, heal them overnight CI pipeline gate — run heal loop automatically before merge Junior devs — let the agent fix the test red before code review

FlagDescriptionDefault
--heal-frameworkjest | vitest | playwright | pytestjest

RAG — Index Codebase

▾

⚙️ Under the Hood
STEP 01
Walk directory tree, collect 25+ file types
Scans .ts, .tsx, .js, .py, .go, .java, .md, .yaml and more. Excludes node_modules/dist/coverage/.git.

Chunk files with overlapping windows

Local embeddings — no external API calls, fully offline. Persistent vector store in .rag_index/.

New team member — index the codebase and ask "how does auth work?" Large monorepos — find where a pattern is used across 500+ files instantly

A11y Healer
Auto-detect and fix all WCAG 2.1 AA violations across your project
▾
python main.py --heal-a11y ./src --a11y-dry-run # preview only
⚙️ Under the Hood
STEP 01
Collect all frontend files (.tsx, .jsx, .html, .vue)
Walks the project directory, skips node_modules/dist/build/coverage.
STEP 02
Send each file to Claude with WCAG 2.1 AA checklist
Checks: missing alt, unlabelled inputs, icon buttons without aria-label, onClick on div without role="button", missing aria-live, landmark elements, focus traps on modals.
STEP 03
Claude rewrites the file with fixes inline
Every change annotated with // a11y: <reason>. Logic, styling, and test IDs are never touched.
STEP 04
Preview diff → confirm → write to disk
Shows which files changed. --a11y-dry-run skips the write step entirely for a safe preview.

MCP Agent — Claude + Model Context Protocol
Connect any MCP server (stdio / SSE / HTTP) and run a full Claude agentic loop with its tools
▾
python main.py --mcp 'npx -y @modelcontextprotocol/server-filesystem .' --mcp-task "List all exported components in src/"
⚙️ Under the Hood
STEP 01
Parse server string into transport config
Detects transport from the string: stdio (any executable), SSE (URL ending in /sse), or Streamable HTTP (other HTTP URL). Constructs MCPServerConfig with command/args or URL accordingly.
STEP 02
Open MCP session + discover tools
Opens an async session via the mcp Python SDK. Calls list_tools() on the server — returns names, descriptions, and JSON Schema input schemas for every tool. Optionally also lists resources.
STEP 03
Convert MCP tools → Claude API tool schemas
Maps each MCPTool to Anthropic's tool format (name, description, input_schema). Server-name prefix prevents collisions when multiple servers expose same-named tools (e.g. filesystem__read_file).
STEP 04
Claude decides which tools to call
The task + all tool schemas are sent to Claude. Claude reasons over the task and emits tool_use blocks specifying which tool to call and with what arguments.
STEP 05
Execute tool calls via MCP server
Each tool_use block is routed to the correct MCP server's call_tool() method. Results (text content blocks) are collected and returned as tool_result messages.
STEP 06
Iterative loop until task complete
Tool results are appended to the message history and sent back to Claude. Claude continues calling tools or producing a final end_turn response. Hard cap at 20 iterations to prevent runaway loops.
💡 Use Cases
Explore a codebase: list components, read files, summarise architecture Query a database via MCP SQL server without writing SQL manually Run Git operations (log, diff, branch) through Claude's reasoning Connect to any third-party MCP server (Stripe, GitHub, Slack, Notion…) Chain multiple MCP servers in a single agentic session
🏹 Output
Tool call trace (console) Claude final answer Token + cost summary
